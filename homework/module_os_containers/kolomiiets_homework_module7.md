# Домашнє завдання (модуль 7) — Контейнери як процеси

Обраний варіант: **A — Аналіз життєвого циклу контейнера**.

Система: Ubuntu 24.04.4 LTS, Docker 29.4.2
Застосунок: вбудований HTTP-сервер Python (`python -m http.server`).

---

## 1. Запуск

```bash
docker run -d --name hw7 -p 8080:8080 python:3-alpine python -m http.server 8080
```

Перевірив, що працює (запит з хоста):

```bash
curl -s -o /dev/null -w "HTTP %{http_code}\n" http://localhost:8080/
```

```
HTTP 200
```

## 2. Процес: який PID 1 усередині контейнера

```bash
docker exec hw7 ps -o pid,ppid,args
```

```
PID   PPID  COMMAND
    1     0 python -m http.server 8080
    7     0 ps -o pid,ppid,args
```

PID 1 усередині — це **сам застосунок** `python -m http.server 8080`.

**Чому саме він:** контейнер — це не «віртуалка», а звичайний Linux-процес у власних namespace'ах (PID, NET, MNT…). Процес, який Docker запускає як головну команду контейнера (`CMD`/аргументи `docker run`), стартує першим у новому PID-namespace, тому отримує PID 1. На хості це звичайний процес зі звичайним PID:

```bash
docker inspect --format '{{.State.Pid}}' hw7
```

```
3432318
```

Тобто PID 1 усередині = PID 3432318 на хості. Один і той самий процес, просто інша «нумерація» через PID-namespace.

## 3. Завершення: `docker stop` і сигнали

```bash
docker stop hw7
```

Заміряв час і код виходу:

```
Elapsed (wall clock) time: 0:10.16
ExitCode=137
```

**Що відбувається:**
- `docker stop` спершу надсилає процесу PID 1 сигнал **SIGTERM** («ввічливо завершись»).
- Чекає grace-період (за замовчуванням **10 секунд**).
- Якщо процес за цей час не завершився — надсилає **SIGKILL** (примусове вбивство, його неможливо перехопити).
- `ExitCode=137` = `128 + 9` → процес убито сигналом **9 (SIGKILL)**.

Цікавий момент: навіть «звичайний» `python -m http.server` тут провисів усі 10 секунд і вийшов через SIGKILL. Причина — **особливість PID 1 у Linux**: для процесу з PID 1 ядро ігнорує сигнали, для яких не встановлено власний обробник. `http.server` не ставить обробник SIGTERM, тому як PID 1 він його просто проігнорував, і Docker довелось добивати його через SIGKILL.

### Що буде, якщо процес ігнорує сигнал — окремий приклад

Запустив контейнер, який явно ігнорує SIGTERM (`trap "" TERM`):

```bash
docker run -d --name hw7ig python:3-alpine \
  sh -c 'trap "" TERM; echo started; while true; do sleep 1; done'
docker exec hw7ig ps -o pid,args
```

```
PID   COMMAND
    1 sh -c trap "" TERM; echo started; while true; do sleep 1; done
    8 sleep 1
```

```bash
docker stop hw7ig    # заміряв час
```

```
stop took 10s
ExitCode=137   (137 = 128+9 = SIGKILL)
```

Тобто: процес проігнорував SIGTERM → Docker почекав 10 с grace-періоду → надіслав SIGKILL → код 137. SIGKILL заблокувати неможливо, тому контейнер усе одно завершився.

> (Grace-період можна змінити: `docker stop -t <сек>`.)

## 4. Логи: `docker logs`

```bash
docker logs hw7
```

```
172.17.0.1 - - [22/Jun/2026 18:03:59] "GET / HTTP/1.1" 200 -
```

**Звідки беруться логи:** застосунок усередині контейнера пише у **stdout/stderr**. Docker перенаправляє ці потоки PID 1 у свій log-driver (за замовчуванням `json-file` на хості), а `docker logs` просто читає цей файл. Тому в логах видно те, що сервер вивів у консоль — у цьому випадку рядок HTTP-запиту. Адреса `172.17.0.1` — це шлюз bridge-мережі `docker0`, тобто запит «прийшов» з хоста через port mapping `-p 8080:8080`.

## Підсумок: що визначає життя контейнера і чому він завершився

- **Контейнер живе рівно стільки, скільки живе його процес PID 1.** Як тільки PID 1 завершується — контейнер зупиняється (статус `Exited`). Це не «середовище, що працює саме по собі», а один головний процес у наборі namespace'ів + cgroups.
- Контейнер завершився, бо ми зробили `docker stop`: Docker надіслав SIGTERM процесу PID 1, той його не обробив (особливість PID 1), тому після 10 с grace-періоду Docker добив його **SIGKILL** → `ExitCode 137`.
- **Висновок своїми словами:** контейнер = процес + ізоляція. PID 1 усередині — це і є застосунок; його сигнальна поведінка (обробляє він SIGTERM чи ні) напряму визначає, чи зупиниться контейнер «чисто» за 1 с, чи його доведеться вбивати через 10 с.
